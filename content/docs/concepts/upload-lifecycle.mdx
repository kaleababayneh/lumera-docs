## Overview

A Cascade upload is not a single point-to-point transfer. It is a multi-phase state transition bridging the Lumera control-plane and the Supernode data-plane. The phases are: **Preparation**, **On-Chain Registration**, and **Execution (SN-API Handoff)**.

<Callout type="info">
While our official SDKs (TypeScript, Rust, Go) abstract this into a single `uploadFile` method, understanding the underlying mechanisms is critical for Protocol Researchers and custom Integrators.
</Callout>

---

## Phase 1: Cryptographic Preparation (Offline)

Before any network request is made, deterministic proofs must be generated entirely client-side.

1. **Payload Hashing**: The raw file payload is converted into bytes. A **BLAKE3 hash** is computed over the entire buffer. This deterministic hash serves as the immutable on-chain identifier.
2. **RaptorQ Layout Generation**: The payload size and network parameters are used to generate an erasure-coding **layout** map. This layout details the exact symbol and block splits.
3. **Deterministic Layout IDs**: From the layout map, deterministic `layout_ids` are generated using a specific nested hashing algorithm involving base64 and zstd compression.
4. **ADR-036 Signatures**: The client signs two distinct payloads using the standard Cosmos [ADR-036 `signArbitrary`](https://docs.cosmos.network/main/build/architecture/adr-036) spec:
   - The compacted layout map (proving the execution parameters).
   - The `data_hash` (proving ownership during the REST upload phase).

---

## Phase 2: On-Chain Registration (`MsgRequestAction`)

Once the proofs are generated, the client initializes the state machine on the Lumera blockchain.

### Broadcasting the Transaction
The client constructs a `/lumera.action.v1.MsgRequestAction` transaction containing the `data_hash`, `layout_ids`, and payload size. The client also escrows the calculated storage fee in `ulume`.

### State Transition: `PENDING`
When the block is minted:
1. The chain validates the transaction and fee.
2. The `x/action` module sets the file's state to `PENDING`.
3. An `action_registered` event is emitted, containing the globally unique `action_id`.

```typescript
// Typical Developer Abstraction (Execution Layer)
const result = await client.Cascade.uploader.registerAction({ fileBytes, dataHash }, ...);
// Returns the emitted `action_id`
```

---

## Phase 3: SN-API Handoff (Data Plane)

With the `action_id` confirmed on-chain, the client shifts entirely to the off-chain REST gateway.

### Multipart Transfer
The client sends a `POST /api/v1/actions/cascade` request to the SN-API.
The payload must strictly contain:
- The `action_id` proving on-chain registration.
- The **ADR-036 auth signature** of the `data_hash` (computed in Phase 1).
- The raw byte buffer.

### Supernode Ingestion & Encoding
Upon receiving the payload, the target Supernode:
1. Verifies the ADR-036 signature to prevent spam.
2. Queries the `x/action` module to confirm the `action_id` exists and is `PENDING`.
3. Verifies the BLAKE3 hash matches the raw bytes.
4. Distributes the chunks across the Supernode peer mesh.

### Final State Transition: `DONE`
Once redundancy thresholds are met natively in the P2P mesh, the ingestion Supernode broadcasts a `/lumera.action.v1.MsgFinalizeAction`. This completes the lifecycle, toggling the on-chain state to `DONE` and freeing the escrowed funds to the storage providers.

## Developer Error States

If your implementation fails, refer to these SDK abstraction states:

| State | Phase | Meaning |
|--------|-------|---------|
| `sdk:registration_failure` | Registration | The `MsgRequestAction` transaction failed or ran out of gas. |
| `sdk:upload_failed` | Handoff | The transfer to SN-API timed out or failed signature verification (ADR-036 issue).|
| `sdk:processing_timeout` | Handoff | Supernodes received it but failed to encode it within timeframe. |
| `sdk:supernodes_unavailable` | Handoff | Rejection at the SN-API load balancer layer. |
