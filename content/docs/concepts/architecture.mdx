## System Overview

Cascade is designed around a strict separation of **control-plane** (the Lumera blockchain state machine) and **data-plane** (the Supernode network). This design ensures that the consensus engine remains fast and lightweight, while large object payloads are deterministic and handled entirely off-chain by specialized nodes.

## State Machine: The `x/action` Module

The core of Cascade's integration with the Cosmos SDK is the `x/action` module. This module tracks the lifecycle of files uploaded to the network, handles the escrowing of storage fees, and defines the deterministic states for off-chain actions.

### State
Actions in the `x/action` module transition through following strict state machine enums:
1. `PENDING`: The action is registered on-chain and awaiting Supernode processing.
2. `PROCESSING`: Supernodes have acknowledged the data and are performing RaptorQ encoding.
3. `DONE`: The encoded chunks are distributed across the network and data is permanently available.
4. `FAILED`: The file was un-processable or corrupted before redundancy.

### Messages (State Transitions)

The module exposes specific protobuf `Msg` types to transition these states.

#### `MsgRequestAction`
Broadcasted by the client (Application Developer) to initialize an upload.
```protobuf
message MsgRequestAction {
  string creator = 1;       // Sender address
  string action_type = 2;   // Identifier, strictly: "ACTION_TYPE_CASCADE"
  bytes  data_hash = 3;     // Deterministic BLAKE3 hash of the file payload
  uint64 data_size = 4;     // File size in bytes
  repeated string rq_ids = 5; // Deterministic RaptorQ layout IDs 
  string app_pubkey = 6;    // Optional: Used for ICA-based actions
}
```

#### `MsgFinalizeAction`
Broadcasted by the Supernode network after they successfully ingest, encode, and distribute the file. This transitions the action to `DONE` and releases the escrowed fee.

## Off-Chain Execution Layer: Supernodes

Supernodes are specialized network actors running parallel infrastructure alongside their standard Lumera validators. They are responsible for solving the **Data Availability (DA)** component of the `x/action` requests.

### Core Responsibilities
- **Ingestion**: Receiving payloads directly from users via the SN-API.
- **Encoding**: Slicing payloads with RaptorQ erasure coding algorithms.
- **Distribution**: Propagating chunks across the independent Supernode P2P mesh.
- **Serving**: Reconstructing the raw file when requested by a client.

<Callout type="info">
Hardware requirements for running a Supernode strictly exceed base validator requirements due to heavy bandwidth and SSD read/write profiles. Refer to the [Supernode Overview](/docs/supernodes/overview) for exact specs.
</Callout>

### SN-API (Supernode REST Gateway)
The SN-API acts as the execution-layer entry point for clients. It abstracts the complexity of the P2P mesh logic.
- Routes downloads and uploads to deterministic Supernode peers.
- Tracks localized task statuses via long-polling or Server-Sent Events (SSE).
- Secures all operations by relying on on-chain ADR-036 signatures.

## Fee & Security Economics

### One-Time Storage Fee
Storage fees are charged natively in `ulume` and are determined dynamically by on-chain parameters based on the `data_size`. 

```typescript
// Query the target fee before building the transaction
const fee = await client.Blockchain.getActionFee(fileSizeInBytes);
```

The protocol relies on a burn-and-distribute tokenomic model. The fee is escrowed upon `MsgRequestAction`. Upon `MsgFinalizeAction`, 20% of the fee is burned (creating deflationary pressure), and 80% is distributed to the Supernodes guaranteeing the chunks.

### Security Guarantees
- **Data Integrity**: Enforced cryptographically. The BLAKE3 hash stored on-chain in `x/action` must exactly match the data served by the Supernode mesh.
- **Authentication**: All interactions with the SN-API require a valid [ADR-036](https://docs.cosmos.network/main/build/architecture/adr-036) message signature, cryptographically linking the caller to the transaction creator.
- **Censorship Resistance**: Due to RaptorQ redundancy, a file remains fully valid and reconstructable even if a massive threshold of Supernodes drop offline.
