## System Overview

Cascade is designed around a strict separation of **control-plane** (blockchain) and **data-plane** (Supernode network). This architecture ensures that on-chain storage remains efficient while large files are handled off-chain by specialized infrastructure.

## Components

### Lumera Blockchain

The Lumera chain is an application-specific Layer-1 built with Cosmos SDK and CometBFT. It handles:

- **Action registration**: `MsgRequestAction` records file metadata (hash, size, layout IDs) on-chain
- **Fee escrow**: Creators pay a one-time storage fee in `ulume` that is escrowed until the action is finalized
- **State machine**: Actions transition through defined states — `PENDING` → `PROCESSING` → `DONE` → `APPROVED` (or `REJECTED` / `FAILED` / `EXPIRED`)
- **Action queries**: Any client can look up an action by ID to verify storage status
- **Supernode compliance**: The [Audit module](/docs/concepts/architecture#audit-module) enforces metrics reporting and penalizes non-compliant Supernodes via epoch-based monitoring

### Supernodes

Supernodes are validators that additionally provide storage and compute capacity to the network. They are responsible for:

- **Receiving file uploads** via the SN-API
- **Encoding files** with RaptorQ erasure coding
- **Storing encoded chunks** with redundancy guarantees
- **Serving file downloads** by reconstructing the original data from available chunks
- **Reporting metrics** every epoch (~400 blocks) to prove compliance
- **Peer observations** — monitoring other Supernodes' port connectivity as part of the audit protocol

Hardware requirements for running a Supernode are detailed in the [Supernode Overview](/docs/supernodes/overview#hardware-requirements).

### SN-API (Supernode API)

The SN-API is a REST/gRPC gateway running on each Supernode (port 8002) that abstracts the complexity of the Supernode mesh:

- Receives file uploads via multipart form data
- Routes download requests to the appropriate Supernodes
- Provides task status tracking via polling and Server-Sent Events (SSE)
- Handles authentication via ADR-036 wallet signatures

Public SN-API endpoints:

| Network | Endpoint |
|---------|----------|
| Testnet | `https://snapi.testnet.lumera.io` |
| Mainnet | `https://snapi.lumera.io` |

### Action Module

The Action module (`x/action`) manages the lifecycle of Cascade storage and Sense verification actions. It supports two action types:

| Action Type | Purpose |
|-------------|---------|
| `ACTION_TYPE_CASCADE` | Decentralized file storage |
| `ACTION_TYPE_SENSE` | Data fingerprinting and verification |

Three core messages drive the action lifecycle:

| Message | Sender | Description |
|---------|--------|-------------|
| `MsgRequestAction` | Creator | Registers a new action on-chain with file metadata and escrows the fee |
| `MsgFinalizeAction` | Supernode | Submitted by the top eligible Supernodes after processing is complete |
| `MsgApproveAction` | Creator | Optional approval step to confirm the action result |

Key parameters (governable):

| Parameter | Description |
|-----------|-------------|
| `base_action_fee` | Base fee charged per action |
| `fee_per_kbyte` | Additional fee per kilobyte of data |
| `super_node_fee_share` | Percentage of fees distributed to Supernodes (default: 90%) |
| `foundation_fee_share` | Percentage of fees allocated to the foundation (default: 10%) |
| `max_actions_per_block` | Maximum actions processed per block |
| `expiration_duration` | Time window for Supernodes to finalize an action before it expires |

### Audit Module

The Audit module (`x/audit`) provides epoch-based monitoring and enforcement for Supernodes:

- **Epochs**: Configurable block intervals (default: 400 blocks). At each epoch boundary, an `EpochAnchor` is persisted containing a seed, active Supernode set, and target assignments.
- **Self-reporting**: Each Supernode must submit one `MsgSubmitEpochReport` per epoch containing host metrics (CPU, RAM, disk, version, ports) and peer observations.
- **Enforcement**: At epoch end, Supernodes that fail to report, report non-compliant metrics, or are observed by peers as having closed ports are moved from `ACTIVE` to `POSTPONED` state.
- **Recovery**: A `POSTPONED` Supernode can return to `ACTIVE` by submitting a compliant self-report and receiving at least one peer confirmation that all required ports are open.

## Data Flow

### Upload Path

1. **Client** computes the BLAKE3 hash of the file
2. **Client** generates a RaptorQ layout and derives deterministic layout IDs (LEP-1)
3. **Client** signs the layout with ADR-036 (`signArbitrary`)
4. **Client** broadcasts `MsgRequestAction` to the chain (simulates gas, signs, broadcasts)
5. **Chain** validates the message, escrows the fee, and emits an `action_registered` event with the `action_id`
6. **Client** sends the file to SN-API (`POST /api/v1/actions/cascade`) with the `action_id` and auth signature
7. **SN-API** encodes the file with RaptorQ and distributes chunks across the Supernode mesh
8. **Client** polls task status until terminal state (`sdk:completed` or `sdk:failed`)
9. **Supernode** broadcasts `MsgFinalizeAction` to transition the action to `DONE`

### Download Path

1. **Client** signs the `action_id` with ADR-036 for authentication
2. **Client** requests download via `POST /api/v1/actions/cascade/{action_id}/downloads`
3. **SN-API** gathers chunks from Supernodes and reconstructs the file using RaptorQ decoding
4. **Client** monitors progress via SSE (`GET /api/v1/downloads/cascade/{task_id}/status`)
5. **Client** streams the file via `GET /api/v1/downloads/cascade/{task_id}/file`

## Fee Model

Cascade storage fees are calculated based on file size and current network parameters:

```ts
const fee = await client.Blockchain.getActionFee(fileSizeInBytes);
// Returns: { fee: string } in ulume
```

The fee is paid once at action registration time. There are no recurring charges. When the action is finalized, the escrowed fee is distributed as follows:

| Recipient | Share | Description |
|-----------|-------|-------------|
| Supernodes | 90% | Distributed to participating Supernodes that processed the action |
| Foundation | 10% | Allocated to the Lumera foundation for protocol development |

These percentages are governance parameters (`super_node_fee_share` and `foundation_fee_share`) and can be updated via on-chain governance proposals.

## Security Model

- **Data integrity**: BLAKE3 hashes are stored on-chain and verified during download reconstruction
- **Authentication**: All SN-API operations require ADR-036 wallet signatures
- **Data availability**: RaptorQ erasure coding ensures files can be reconstructed even if a significant fraction of Supernodes go offline
- **Immutability**: Once an action is registered on-chain, the associated data hash cannot be modified
- **Supernode compliance**: The Audit module enforces metrics reporting, port availability, and hardware minimums — non-compliant Supernodes are automatically postponed
