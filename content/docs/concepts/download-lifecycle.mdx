## Overview

Unlike standard HTTP endpoints, downloading a file from Cascade requires proof of protocol state. You do not define a specific URL for a file; instead, you request execution against the on-chain **action ID** generated during the upload phase. The download process involves cryptographic authentication, task delegation via the SN-API, and data reconstruction.

---

## Step 1: Client Authentication (ADR-036)

To prevent arbitrary bandwidth consumption and secure private files, all download requests must be authenticated. The client signs the exact string of the on-chain `action_id` using the standard [ADR-036](https://docs.cosmos.network/main/build/architecture/adr-036) signature scheme. 

```typescript
// Typical Developer Abstraction
const signature = await signer.signArbitrary(chainId, address, actionId);
```

This ensures that:
- **Public files** can be downloaded by any valid Lumera wallet signature.
- **Private files** are strictly gate-kept by the Supernodes, which enforce that the signing `address` matches the `creator` address or an authorized delegated wallet on the original `MsgRequestAction`.

---

## Step 2: Request Execution via SN-API

Rather than opening a static file stream, the SDK initializes a task on the Supernode gateway.

The client sends a generic `POST` request to `/api/v1/actions/cascade/{action_id}/downloads`, passing the ADR-036 signature in the body payload.

Upon receiving the request, the Supernode:
1. Validates the signature cryptographically.
2. Cross-references the `action_id` against the Lumera `x/action` module to confirm its state is `DONE`.
3. Issues a localized `task_id` for tracking.

---

## Step 3: Supernode Fetching & Reconstruction

Once the task is initialized, heavy lifting occurs off-chain within the data-plane.

The target Supernode acts as an aggregator:
- It queries its P2P peers to locate the distributed RaptorQ chunks corresponding to the `action_id`.
- It downloads a sufficient number of these chunks (due to erasure coding, it does not need all of them, just enough to pass the restoration threshold).
- It reconstructs the original payload in memory.

During this period, the client tracks progress via **Server-Sent Events (SSE)** on `/api/v1/downloads/cascade/{task_id}/status`.

---

## Step 4: Stream the Decoded Payload

Once the Supernode fully restores the file, the task transitions to `completed`. The client can now fetch the native binary file stream directly from `/api/v1/downloads/cascade/{task_id}/file`.

The SDK returns a `ReadableStream<Uint8Array>` (in TS) or native streams in Rust/Go, allowing developers to reconstruct files dynamically without consuming massive memory overheads on the local device.
