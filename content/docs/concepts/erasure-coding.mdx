## Why Erasure Coding?

Standard redundancy (like 5x replication) scales poorly for large datasets. If the network needs resilience against 50% node dropout, raw replication doubles storage costs. 

Cascade solves this using **RaptorQ** (RFC 6330), an advanced fountain code. It allows the protocol to hit massive redundancy thresholds with only ~1.5x on-disk overhead.
- An ingested file is mathematically sliced into $N$ symbols. 
- It can be fully completely reconstructed using *any* $K$ subset of those symbols, assuming $K < N$.
- Probability of decoding failure drops to mathematical zero when the network holds just a few percentage points of parity overhead.

## Systematic Upload-Download Flow

### Storage Ingestion (Supernode DA)

<Diagram name="erasure-coding" />

When a client broadcasts a payload to the SN-API:
1. The deterministic Rust/WASM RaptorQ implementation generates an execution **layout**—defining block sizing and symbol boundaries.
2. The payload is chunked into encoded symbols.
3. These symbols are pushed rapidly across the independent P2P mesh connecting active Supernodes. 

### Data Retrieval

When a download task is requested:
1. The receiving Supernode pings its closest peers.
2. It fetches chunks until the $K$ threshold is met.
3. The RaptorQ decoder computes the original payload entirely in memory.
4. The file's BLAKE3 hash is compared cryptographically against the hash locked in the `MsgRequestAction` of the `x/action` module.

## LEP-1: Deterministic Symbol Identifiers

To prevent data corruption, every RaptorQ chunk must possess a verifiable identifier. LEP-1 (Lumera Enhancement Proposal 1) defines the algorithm for this derivation. 

### Layout Parameters
When formulating a block layout, parameters are fetched directly from the `x/action` genesis or governance state (e.g., `rq_ids_max`). 

### ID Derivation Protocol Specifications
Each chunk ID (`layout_id`) is derived deterministically using the network's Go specification:

```text
For i = 0 to rq_ids_max - 1:
  counter = rq_ids_ic + i
  input = Base64(compacted_layout_json) + "." + Base64(adr036_signature) + "." + String(counter)
  compressed = zstd_compress(input, level=3)
  hash = BLAKE3(compressed)
  layout_id = Base58(hash)
```

Why force derivation?
- **Immutability**: A layout's geometry cannot be altered without failing verification.
- **Protocol Safety**: The `x/action` module natively checks the deterministic roots before acknowledging `MsgFinalizeAction`.

---

## The Index Payload

To aggregate these chunks safely without bloating the blockchain's block space, the client generates a JSON indexing payload:

```json
{
  "version": 1,
  "layout_ids": ["7K9xQ2b...", "3Jm2pNd...", "..."],
  "layout_signature": "base64-encoded ADR-036 payload"
}
```

The stringified `layout_ids` array is packed into the actual `MsgRequestAction` protobuf payload.
