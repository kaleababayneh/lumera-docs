This is the reusable primitive the rest of this section builds on. Whatever you're shipping вҖ” an NFT marketplace, a DAO, a derivatives venue, prediction markets вҖ” the shape is the same: **Injective holds state and pointers, Cascade holds the bytes, and an `action_id` ties them together**. The only thing that varies is *who signs the Cascade write*.

## The `action_id` pointer pattern

Cascade's unit of storage is an **action**. A `MsgRequestAction` on Lumera registers an artifact's metadata and returns a numeric `action_id`; the file bytes are streamed to a Supernode against that ID. Once stored, the artifact is retrievable forever by `GET /download/<action_id>` against any cascade-aware gateway. See [How Cascade Works](/docs/getting-started/how-cascade-works) for the full lifecycle.

Inside your Injective contract, every Cascade artifact is referenced by a single value: that `action_id`, stored as a `String` in CosmWasm state.

```rust title="a contract storing a Cascade pointer"
pub struct StoredArtifact {
    pub cid: String,         // вҶҗ Cascade action_id, e.g. "15823"
    pub submitter: Addr,
    pub block_height: u64,
}
```

That single `String` is the entirety of the cross-chain coupling. The contract enforces only *that the field is non-empty* вҖ” it does not enforce *that the artifact resolves*, because it cannot read across chains synchronously. Resolution and integrity are read-time concerns, handled by an indexer or the frontend, off the consensus path.

## Two ways to write to Cascade

There are two production patterns for getting bytes onto Cascade from an Injective context. They are **not** mutually exclusive вҖ” many deployments use both, choosing per artifact type.

### Pattern A: user-signed (the user signs the Lumera tx)

The user holds both an `inj1вҖҰ` and a `lumera1вҖҰ` address in their wallet. The dApp signs **two transactions** for a single action: first a Lumera tx that uploads to Cascade and returns an `action_id`, then an Injective tx that submits that `action_id` to your contract.

### Pattern B: server-signed via cascade-api (recommended)

A backend holds one funded Lumera signing key and uploads on behalf of users. The user signs **only** the Injective transaction; the upload is a server-side HTTP call. The simplest way to run this is to forward to the **Lumera-operated cascade-api at `api.lumera.help`**, which holds the Lumera key, signs `MsgRequestAction`, and pays the `ulume` fee for you.

Choose this when **UX matters**: one signature instead of two, no Lumera address required for the user, and a single rate-limited signing key you can monitor and rotate. This is the path the [Inscribe reference implementation](/docs/injective/inscribe) uses end-to-end, and it's the fastest way to ship.

The trade-off is provenance: every artifact is signed by the operator's key, not the user's. If you need on-chain proof that *user X* uploaded *artifact Y*, record the binding on Injective вҖ” have your contract store `(user, action_id)` so the pairing is the on-chain proof rather than the Lumera signature.

If you'd rather hold the Lumera key yourself instead of using the hosted cascade-api, the equivalent with `@lumera-protocol/sdk-js` is:

```ts title="server-side upload with your own Lumera key"
import { createLumeraClient } from "@lumera-protocol/sdk-js";

// One funded Lumera key uploads on behalf of users.
const client = await createLumeraClient({
  preset: "testnet",
  signer,
  address: serverLumeraAddress,
  gasPrice: "0.025ulume",
});

const result = await client.Cascade.uploader.uploadFile(file, {
  fileName: "artifact.json",
  isPublic: true,
  expirationTime: String(Math.floor(Date.now() / 1000) + 24 * 60 * 60),
});
// result.action_id  вҶ’ 15823
```

### Pattern C: contract-driven via ICA (advanced)

There is a third pattern, where a CosmWasm contract on Injective writes to Cascade **autonomously** over IBC: the contract packs a `MsgRequestAction`, sends it across the IBC channel to a registered [Interchain Account](/docs/concepts/interchain-accounts) on Lumera, and the ICA executes it.

<Callout type="warn">
**This is not on the hot path, and you should not reach for it first.** An earlier prototype of the reference implementation dispatched Cascade writes this way via [`cw-ica-controller`](https://github.com/srdtrk/cw-ica-controller) and waited for a Hermes relayer. That works when the relayer is healthy but degrades to multi-minute waits when it is backed up. The current architecture has no IBC step вҖ” it uses Pattern B. Reserve Pattern C for canonical, contract-only artifacts where there is genuinely no user in the loop and the *contract itself* must be the signer.
</Callout>

**Rule of thumb:** use **A** when on-chain provenance to the user's key is required, **B** for everything else (and as your default), and consider **C** only for contract-deterministic canonical writes where IBC latency is acceptable.

## Reading from Cascade

Reading is independent of the write pattern. Once an artifact has an `action_id`, anyone can fetch it, with no authentication, from any cascade-aware gateway:

| Endpoint | Auth | Returns |
|---|---|---|
| `GET https://api.lumera.help/download/{action_id}` | none | raw bytes, `Content-Type` sniffed |
| `GET <your-backend>/cascade/{action_id}` | your choice | raw bytes, ideally with an immutable cache header |
| Lumera SDK | wallet signature | reconstructed bytes streamed |

For dApp reads you want the public HTTPS gateway. Because Cascade content is content-addressed, you can cache it with `Cache-Control: public, max-age=31536000, immutable` вҖ” once you have a CID's bytes, they are correct forever, so there is no invalidation strategy to design.

## Wiring the pointer into your contract

The Cascade integration on the CosmWasm side is small. Three conventions cover almost everything:

**1. Accept a CID at write time, validate non-empty:**

```rust
pub fn exec_submit(
    deps: DepsMut, env: Env, info: MessageInfo, cid: String,
) -> Result<Response, ContractError> {
    if cid.trim().is_empty() {
        return Err(ContractError::InvalidCid("empty cid".into()));
    }
    // ... append to state, emit event with cid as attribute
}
```

**2. Emit the CID as a structured event attribute so an indexer can pick it up:**

```rust
Ok(Response::new()
    .add_attribute("action", "submit")
    .add_attribute("cid", cid)
    .add_attribute("submitter", info.sender))
```

**3. Expose CIDs in queries so a read-only client can drive a viewer without an indexer:**

```rust
#[returns(Vec<StoredArtifact>)]
ListArtifacts {},
```

That's the essential contract-side surface: the contract does not download from Cascade, does not verify the artifact, does not even care that it's a Cascade CID вҖ” it holds a string, and verification and resolution belong to the indexer and the frontend. For production designs, [Build your own integration](/docs/injective/build-your-own#contract-side-conventions) adds two more conventions вҖ” modelling the workflow as a state machine, and locking funds atomically with the CID write.

## Funding the writes

Cascade writes are paid in `ulume`. Which account pays depends on the pattern:

| Account | What it pays for | How to fund |
|---|---|---|
| The user's `lumera1вҖҰ` (Pattern A) | Each `MsgRequestAction` they sign | [Faucet](/docs/faucet) |
| Your service's Lumera key (Pattern B) | Each `MsgRequestAction` your service signs on behalf of users | [Faucet](/docs/faucet); monitor balance and alert before exhaustion |
| The ICA's `lumera1вҖҰ` (Pattern C) | Each Cascade write the contract triggers | Top up over IBC from your Injective treasury |

A Cascade upload costs roughly **10вҖ“15k `ulume`** on testnet. At a fresh faucet balance of `1,000,000 ulume`, that's ~70 uploads before a refill вҖ” plan capacity accordingly.
