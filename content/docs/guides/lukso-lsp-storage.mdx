## Overview

[Lukso](https://lukso.network) is an EVM L1 whose flagship feature — Universal Profiles — leans on **off-chain JSON metadata** referenced from on-chain `ERC725Y` storage via an LSP-2 [`VerifiableURI`](https://docs.lukso.tech/standards/metadata/lsp2-json-schema). Today that metadata is almost always pinned to IPFS through Pinata or Infura, which means it lives only as long as someone keeps paying the pinning bill.

Cascade is a strict upgrade for that exact slot: pay once, store forever, with on-chain proof of storage. This guide walks through wiring Cascade in as the storage layer for the three most common LSP metadata keys:

- **LSP-3** `LSP3Profile` — Universal Profile metadata (name, description, images)
- **LSP-4** `LSP4Metadata` — collection-level metadata for LSP-7 / LSP-8 tokens
- **LSP-8** per-token metadata — NFT metadata addressable by `tokenId`

<Callout>
Lukso has no IBC client, so this is a **storage / metadata** integration, not a token bridge. `LYX` stays on Lukso, `LUME` stays on Lumera. The integration runs entirely off-chain through a JS library.
</Callout>

## How the Pieces Fit

Every off-chain LSP metadata pointer is a `VerifiableURI`:

```text
0x0000 <methodId 4B> 0x0020 <keccak256 hash 32B> <URI bytes…>
```

The hash binds the URI's payload — readers retrieve the bytes, recompute the hash, and compare. **The URI scheme itself is opaque to the standard**, so any URL that returns the original bytes works: `ipfs://`, `https://`, even `cascade://` if a resolver is registered.

Cascade's existing Lukso adapter, [`@lukso/data-provider-cascade`](https://www.npmjs.com/package/@lukso/data-provider-cascade), uploads to a Cascade gateway and returns an `ipfs://<cid>` URI. Standard Lukso readers don't need to know Cascade exists — Cascade is invisible plumbing under a familiar `ipfs://` URI, with the additional guarantee that the bytes are erasure-coded across Supernodes and retrievable forever.

## Install

```bash
npm install @erc725/erc725.js @lukso/data-provider-cascade @lukso/lsp-smart-contracts viem
```

The `@lumera-protocol/sdk-js` is **not** required for this path — `@lukso/data-provider-cascade` does the upload over HTTPS to the Cascade gateway. Reach for the Lumera SDK only when you want to upload from a Cosmos signer (e.g. Keplr) instead of a public gateway API key.

## 1. Build the LSP-3 JSON

The LSP-3 schema is fixed: see [LSP-3 Profile Metadata](https://docs.lukso.tech/standards/metadata/lsp3-profile-metadata).

```ts title="src/lsp3.ts"
export type LSP3Profile = {
  LSP3Profile: {
    name: string;
    description: string;
    links: { title: string; url: string }[];
    tags: string[];
    profileImage: { width: number; height: number; url: string; verification?: { method: string; data: string } }[];
    backgroundImage: { width: number; height: number; url: string }[];
  };
};

export function buildProfile(input: {
  name: string;
  description: string;
  profileImageUrl: string;
}): LSP3Profile {
  return {
    LSP3Profile: {
      name: input.name,
      description: input.description,
      links: [],
      tags: ["lumera", "cascade"],
      profileImage: [
        { width: 512, height: 512, url: input.profileImageUrl },
      ],
      backgroundImage: [],
    },
  };
}
```

## 2. Upload via CascadeUploader

```ts title="src/upload.ts"
import { CascadeUploader } from "@lukso/data-provider-cascade";

const cascade = new CascadeUploader(process.env.CASCADE_API_KEY!);

export async function uploadJSON<T>(value: T): Promise<{ url: string; hash: string }> {
  const blob = new Blob([JSON.stringify(value)], { type: "application/json" });
  const file = new File([blob], "metadata.json", { type: "application/json" });
  // CascadeUploader returns { url: "ipfs://<cid>", hash: "0x…" }
  return await cascade.upload(file);
}

export async function uploadBinary(file: File): Promise<{ url: string; hash: string }> {
  return await cascade.upload(file);
}
```

<Callout type="warn">
The Cascade API key must stay server-side. In a Next.js app, put `CascadeUploader` behind a Route Handler (`app/api/upload/route.ts`) and call it from the client via `fetch`. Never ship the key to the browser.
</Callout>

## 3. Encode the VerifiableURI

`erc725.js` knows the LSP-2 byte layout. Pass the JSON object and its URL; it computes the hash and stitches the bytes together.

```ts title="src/encode.ts"
import { ERC725 } from "@erc725/erc725.js";
import LSP3ProfileSchema from "@erc725/erc725.js/schemas/LSP3ProfileMetadata.json" assert { type: "json" };

export function encodeLSP3(profileAddress: `0x${string}`, json: object, url: string) {
  const erc725 = new ERC725(LSP3ProfileSchema, profileAddress, "https://rpc.mainnet.lukso.network");
  const encoded = erc725.encodeData([
    {
      keyName: "LSP3Profile",
      value: { json, url },
    },
  ]);
  // encoded.keys[0]   → 0x5ef83ad9559033e6e941db7d7c495acdce616347d28e90c7ce47cbfcfcad3bc5
  // encoded.values[0] → 0x00006f357c6a0020<hash><ipfs-url-bytes>
  return { key: encoded.keys[0], value: encoded.values[0] };
}
```

## 4. setData on the Universal Profile

A Universal Profile is an [LSP-0 ERC725Account](https://docs.lukso.tech/contracts/overview/UniversalProfile). The owner (or a controller key) calls `setData(bytes32 key, bytes value)`:

```ts title="src/setData.ts"
import { createWalletClient, custom, parseAbi } from "viem";
import { lukso } from "viem/chains";

const UP_ABI = parseAbi([
  "function setData(bytes32 dataKey, bytes dataValue) external payable",
]);

export async function writeProfile(
  profileAddress: `0x${string}`,
  key: `0x${string}`,
  value: `0x${string}`,
) {
  const client = createWalletClient({
    chain: lukso,
    transport: custom(window.lukso!), // UP browser extension
  });
  const [account] = await client.requestAddresses();
  return client.writeContract({
    account,
    address: profileAddress,
    abi: UP_ABI,
    functionName: "setData",
    args: [key, value],
  });
}
```

End-to-end, the flow looks like this:

```ts
const imageUpload = await uploadBinary(file);             // ipfs://<cid> via Cascade
const profile     = buildProfile({ name, description, profileImageUrl: imageUpload.url });
const jsonUpload  = await uploadJSON(profile);            // ipfs://<cid> via Cascade
const { key, value } = encodeLSP3(upAddress, profile, jsonUpload.url);
await writeProfile(upAddress, key, value);                // setData on the UP
```

## 5. LSP-4 (Collections) and LSP-8 (Per-Token Metadata)

The pattern is identical — only the schema and the contract method change.

**LSP-4 collection metadata** is a single key on the token contract:

```ts
import LSP4Schema from "@erc725/erc725.js/schemas/LSP4DigitalAsset.json" assert { type: "json" };

const erc725 = new ERC725(LSP4Schema, tokenContractAddress, RPC);
const { keys, values } = erc725.encodeData([
  { keyName: "LSP4Metadata", value: { json: lsp4Json, url: cascadeUrl } },
]);
// Then setData(keys[0], values[0]) on the token contract.
```

**LSP-8 per-token metadata** uses `setDataForTokenId` (LSP-8 v0.14+):

```ts
import { parseAbi, toHex, padHex } from "viem";

const LSP8_ABI = parseAbi([
  "function setDataForTokenId(bytes32 tokenId, bytes32 dataKey, bytes dataValue) external",
]);

const tokenIdBytes32 = padHex(toHex(tokenId), { size: 32 });
await client.writeContract({
  account,
  address: nftContract,
  abi: LSP8_ABI,
  functionName: "setDataForTokenId",
  args: [tokenIdBytes32, key, value], // key = LSP4Metadata, value = VerifiableURI
});
```

The metadata JSON now lives on Cascade; the on-chain pointer is small, hash-bound, and survives any pinning service going dark.

## How Retrieval Works

Reading is the inverse — and crucially, Lukso clients don't need a Cascade-aware reader. Any IPFS gateway resolves the `ipfs://<cid>` that Cascade returns:

```ts title="src/read.ts"
import { ERC725 } from "@erc725/erc725.js";
import LSP3ProfileSchema from "@erc725/erc725.js/schemas/LSP3ProfileMetadata.json" assert { type: "json" };

const erc725 = new ERC725(LSP3ProfileSchema, profileAddress, "https://rpc.mainnet.lukso.network", {
  ipfsGateway: "https://api.universalprofile.cloud/ipfs",
});

const data = await erc725.fetchData("LSP3Profile");
// erc725.js: getData → decode VerifiableURI → fetch bytes via gateway → keccak256 check → return JSON
console.log(data.value); // { LSP3Profile: { name, description, profileImage: [...] } }
```

`fetchData` does four things in one call: pulls the on-chain `bytes`, decodes the `VerifiableURI` header, fetches the JSON via the configured `ipfsGateway`, and verifies the keccak256 hash matches the on-chain commitment. If the hash doesn't match, it throws — so a corrupted or substituted gateway response is rejected by construction.

## Why This Is a Strict Upgrade

| | IPFS pinning (Pinata / Infura) | Cascade |
|---|---|---|
| **Persistence** | Pin-dependent — lapsed pin = lost data | Permanent, enforced by protocol |
| **Pricing** | Recurring | Pay once |
| **Redundancy** | None by default | RaptorQ erasure coding across Supernodes |
| **Self-healing** | No | Yes — automatic chunk restoration |
| **Reader compatibility** | Standard IPFS gateways | Standard IPFS gateways (Cascade returns `ipfs://`) |

The reader compatibility row is the important one for a Lukso integration: **Cascade is a drop-in upgrade**. No reader-side changes, no new wallets, no new schemas. The only thing that changes is who's keeping the bytes alive in 2030.
