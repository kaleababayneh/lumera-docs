## Daemon Management

### Process Controls
```bash
# Evaluate runtime block stream
sudo journalctl -fu lumera

# Restart the consensus engine
sudo systemctl restart lumera

# Halt the state machine safely
sudo systemctl stop lumera
```

### Sync & State Inquiries
```bash
# Query the local node's synchronization state
lumerad status 2>&1 | jq '.sync_info | {latest_block_height, catching_up}'

# Query all external network peers natively connected
lumerad status 2>&1 | jq '.node_info.other'
```

## The `x/staking` & `x/distribution` Modules

Use these transactions to manage your Validator's economy. Ensure you replace `lumera-testnet-2` if operating on Mainnet.

### Delegate Additional Stake
Self-delegate more `ulume` to increase your consensus voting power:
```bash
lumerad tx staking delegate $(lumerad keys show validator --bech val -a) 1000000ulume \
  --from=validator \
  --chain-id=lumera-testnet-2 \
  --gas=auto --gas-adjustment=1.3 \
  --fees=10000ulume \
  -y
```

### Withdraw Commission & Validation Rewards
The protocol aggregates rewards per block. Sweep them to your liquid balance:
```bash
lumerad tx distribution withdraw-rewards $(lumerad keys show validator --bech val -a) \
  --from=validator \
  --commission \
  --chain-id=lumera-testnet-2 \
  --gas=auto --gas-adjustment=1.3 \
  --fees=10000ulume \
  -y
```

## Slashing & Unjailing

If your validator drops offline for an extended period, it violates consensus liveness rules. The `x/slashing` module will automatically "jail" your node and burn a fraction of your stake.

1. **Verify your jailing state** by checking your pubkey against the slashing record:
```bash
lumerad query slashing signing-info $(lumerad tendermint show-validator)
```

2. **Wait for the downtime window** to pass.
3. **Broadcast an Unjail transaction** to re-enter the active set:
```bash
lumerad tx slashing unjail \
  --from=validator \
  --chain-id=lumera-testnet-2 \
  --gas=auto --gas-adjustment=1.3 \
  --fees=10000ulume \
  -y
```

## Telemetry & Observability

If you enabled Prometheus in your `config.toml` (`prometheus = true`), CometBFT exposes endpoints for metric scrapping.
- **Port**: `26660`
- **Scrape Target**: `http://localhost:26660/metrics`

You can attach standard Grafana dashboards to visualize block latency, mempool sizing, and peer thresholds.
