## Prerequisites

Before registering your validator state on-chain, ensure:
- Your node daemon (`lumerad`) is running.
- Your node is fully synchronized (`catching_up: false`).

Verify sync status natively:
```bash
lumerad status 2>&1 | jq '.sync_info.catching_up'
# Expected output: false
```

---

## 1. Initialize the Cosmos Keyring

You need a funded wallet to sign the `MsgCreateValidator` transaction. Create an operator wallet in the local `lumerad` keyring.

```bash
lumerad keys add validator
```

You will securely generate a local keyring passphrase and receive your 24-word BIP39 mnemonic. 

<Callout type="error">
**Mnemonic Backup Required**
The 24-word mnemonic is the master seed of your Validator Operator (`valoper`) identity. Copy it strictly offline. It is NEVER recoverable.
</Callout>

### Query Address Formats

Your identity maps to two distinct benchmark addresses:

```bash
# Standard address (for receiving LUME tokens)
lumerad keys show validator -a
# Expected format: lumera1...

# Validator Operator address (for staking/delegation commands)
lumerad keys show validator --bech val -a
# Expected format: lumeravaloper1...
```

---

## 2. Fund the Operator Wallet

To prevent spam, Lumera enforces a minimum self-delegation bound alongside baseline gas fees (`ulume`). 

- Request testnet tokens via the [Lumera Faucet](https://faucet.astrostake.xyz/lumera).
- If the faucet is drained, request funds via the `#testnet-faucet` channel in the Discord.

Verify the balances module via RPC:
```bash
lumerad query bank balances $(lumerad keys show validator -a)
```
*Note: You need at least 1,000,000 `ulume` (1 LUME) for the delegation + ~10,000 `ulume` for gas.*

---

## 3. Construct and Broadcast `MsgCreateValidator`

Cosmos SDK `v0.50` requires the `MsgCreateValidator` payload to be constructed via a strictly formatted JSON file, deprecating the legacy inline CLI flags.

### Capture the Consensus Pubkey
First, identify the exact CometBFT consensus pubkey generated during the initialized `node_setup`:
```bash
lumerad tendermint show-validator
```

### Build the Configuration JSON
Construct the transaction payload locally:
```bash
cat > ~/validator.json <<EOF
{
  "pubkey": $(lumerad tendermint show-validator),
  "amount": "1000000ulume",
  "moniker": "YOUR_MONIKER",
  "commission-rate": "0.10",
  "commission-max-rate": "0.20",
  "commission-max-change-rate": "0.01",
  "min-self-delegation": "1"
}
EOF
```

| Field | Definition |
|-----------|-------------|
| `amount` | The initial bonded self-delegation. |
| `commission-rate` | Initial fraction of validator rewards charged to your delegators (0.10 = 10%). |
| `commission-max-rate` | Absolute ceiling limit for commission (immutable state). |
| `commission-max-change-rate` | Absolute maximum daily fluctuation limit for the commission. |
| `min-self-delegation` | Hard floor self-delegation balance. Dropping below this automatically jails the validator. |

### Broadcast the Transaction
Sign the JSON payload and broadast it to the mempool:

```bash
lumerad tx staking create-validator ~/validator.json \
  --from=validator \
  --chain-id=lumera-testnet-2 \
  --gas=auto \
  --gas-adjustment=1.3 \
  --fees=10000ulume \
  -y
```

<Callout type="warn">
Ensure you use an elevated `--fees` parameter. If the transaction hangs or errors with `insufficient fees`, bump `--fees=20000ulume`.
</Callout>

---

## 4. Verify On-Chain State

Poll the `x/staking` module to verify your node was registered successfully:

```bash
lumerad query staking validator $(lumerad keys show validator --bech val -a)
```

Look for `status: BOND_STATUS_UNBONDED` or `status: BOND_STATUS_BONDED`. If your total delegation is high enough, you will enter the Active Set and begin signing CometBFT blocks.
