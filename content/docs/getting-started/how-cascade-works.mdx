## The Execution Pipeline

The Lumera SDKs abstract the complexities of cross-chain storage into a single operational pipeline. Under the hood, Cascade separates concerns into a **control plane** (the Lumera Cosmos state machine) and a **data plane** (the Supernode DA mesh).

<Diagram name="how-cascade-works" />

When you invoke `client.Cascade.uploader.uploadFile()`, the SDK natively orchestrates five distinct network transitions:

### 1. Cryptographic Preparation & Indexing
The SDK runs entirely offline in your client to prepare the file for decentralized routing.
- Computes a deterministic **BLAKE3 hash** of the raw file data (the `data_hash`).
- Generates a **RaptorQ layout array**, calculating exactly how the Supernodes will erasure-encode the file.
- Signs a cryptographic **Index File** bridging the file to the SDK wallet.

### 2. State Machine Registration (`MsgRequestAction`)
The SDK broadcasts a standard Cosmos state mutation (`/lumera.action.v1.MsgRequestAction`) to the blockchain. 
This payload formally escrows the required `ulume` storage fee and locks the file metadata (`data_hash`, access flags, expiration bounds) into the `x/action` module.

### 3. State Confirmation (`action_id`)
Upon successful block inclusion, the Tendermint consensus engine emits an `action_registered` event. 
The SDK parses this event to extract the permanent **`action_id`**. This serves as the universal identifier for the file lifecycle.

### 4. Direct Node Streaming (SN-API)
Bypassing the blockchain entirely, the SDK queries the active Supernode set and opens a direct POST stream to an SN-API gateway (`/api/v1/actions/cascade`).
The payload requires:
- `action_id`: The verified consensus event identity.
- `signature`: An off-chain (ADR-036) authorization confirming the uploader owns the wallet that registered the action.
- `file`: The raw byte array.

### 5. Finalization (`MsgFinalizeAction`)
The receiving Supernode confirms the data integrity against the `x/action` record, distributes the erasure-coded chunks across the Kademlia DHT, and signs a `MsgFinalizeAction`.
This transaction transitions the `action_id` state from `PENDING` to `DONE`, formally releasing the escrowed funds to the storage providers. Your file is permanently stored.
