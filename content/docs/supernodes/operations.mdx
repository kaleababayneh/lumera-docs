## Supernode State Machine

The on-chain state of your DA pipeline is evaluated algorithmically. 

| Internal State | Execution Definition | Recovery Action |
|-------|---------|--------|
| `SUPERNODE_STATE_ACTIVE` | Operating normally, processing `MsgRequestAction` tasks. | None |
| `SUPERNODE_STATE_DISABLED` | Deregistered manually by the Validator. | Must re-register via `MsgRegisterSupernode`. |
| `SUPERNODE_STATE_STOPPED` | Paused gracefully for maintenance. | Broadcast a `MsgStartSupernode` tx. |
| `SUPERNODE_STATE_PENALIZED` | Validator operator slashed for malicious behavior. | Permanently banned. Cannot re-enter active set. |
| `SUPERNODE_STATE_POSTPONED` | Metrics failure or unresponsiveness. | Auto-recovers once telemetry is successfully ingested. |

## On-Chain Telemetry (LEP-4)

Supernodes maintain the security of the DA tier by self-reporting telemetry back to the base layer.
Every **~400 blocks**, the `sn-manager` executes an automated IBC or native packet containing:
* Current Node Version
* CPU/RAM/NVMe overhead signatures
* Port connectivity states
* Peer counts and uptime

If the `x/supernode` module does not receive this packet within a ~100 block grace period, your validator is transitioned to `POSTPONED` and stops receiving file chunk shards. The daemon handles this reporting completely autonomously.

## Process & Telemetry Management

### Service Process
```bash
# Daemon health and mesh integration checks
sn-manager status
sn-manager check

# Daemon logs
sudo journalctl -fu sn-manager
```

### Versioning & Upgrades
The daemon ships with an over-the-air version manager. If initialized with `--auto-upgrade`, the daemon will seamlessly swap binaries on governed block heights.

To manually configure an upgrade:
```bash
# List all governed releases natively
sn-manager ls-remote

# Build and swap to a new release
sn-manager get v2.4.72
sn-manager use v2.4.72

sudo systemctl restart sn-manager
```

## Lifecycle Transactions

The `x/supernode` module accepts lifecycle mutations. **All commands must run on the core Validator host, signed by the core `valoper` key.**

### Mutate Public Endpoints
If your Supernode's server IPs change, mutate the state endpoint immediately:
```bash
VALOPER=$(lumerad keys show validator --bech val -a)

lumerad tx supernode update-supernode \
  "$VALOPER" "NEW_PUBLIC_IP:4444" "migrated cloud provider" "<SN_ACCOUNT_ADDRESS>" \
  --from validator \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume -y
```

### Pause State (`MsgStopSupernode`)
It is highly recommended to explicitly signal downtime before engaging in extensive hardware maintenance.

```bash
# 1. Broadast the halt signal
lumerad tx supernode stop-supernode $(lumerad keys show validator --bech val -a) "kernel upgrade" \
  --from validator \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume -y

# 2. Safely stop the daemon locally on the Supernode server
sudo systemctl stop sn-manager
```

### Resume State (`MsgStartSupernode`)
Once maintenance concludes, recover your node to `ACTIVE` status.

```bash
# 1. Start the daemon locally on the Supernode server
sudo systemctl start sn-manager

# 2. Broadast the start signal
lumerad tx supernode start-supernode $(lumerad keys show validator --bech val -a) \
  --from validator \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume -y
```

### Deregistration (`MsgDeregisterSupernode`)
Permanently unbind the Supernode from your consensus layer.

```bash
lumerad tx supernode deregister-supernode $(lumerad keys show validator --bech val -a) \
  --from validator \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume -y
```

## Reference Configurations

The daemon injects your config statically at `~/.supernode/config.yml`.

```yaml
supernode:
  key_name: "supernode"
  identity: "lumera1..."
  host: "0.0.0.0"
  port: 4444

keyring:
  backend: "file"
  dir: "keys"
  passphrase_env: "SUPERNODE_PASSPHRASE"

p2p:
  port: 4445
  data_dir: "data/p2p"

lumera:
  grpc_addr: "<VALIDATOR_IP>:9090"
  chain_id: "lumera-testnet-2"

raptorq:
  files_dir: "raptorq_files"
```

<Callout type="error">
**Critical Backup Paths**
If rebuilding the machine, ensure you backup `~/.supernode/config.yml`, the recursive `~/.supernode/keys/` directory, and your original 24-word generated mnemonic. You cannot rebuild the keys without the mnemonic.
</Callout>
