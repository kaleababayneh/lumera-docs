## What is a Lumera Supernode?

A Supernode operates as a compute-intensive companion to a base Lumera validator. While the validator parses the consensus state machine (`x/action`), the Supernode resolves the off-chain execution mechanisms: **Cascade storage DA**, **Sense verification**, and **AI inference streaming**.

<Callout type="error">
**Physical Isolation Required**
Every Supernode must be tethered to a bonded consensus validator, but it **cannot** run on the same hardware. The storage node must run on an isolated server to prevent high I/O blocking the Tendermint consensus engine.
</Callout>

## Architectural Mapping

<Diagram name="supernode-architecture" />

To bridge the on-chain and off-chain execution limits: 
1. The Supernode binds as a local client to the Validator's open `gRPC` port (TCP `9090`). It uses this connection to listen to mempool events and verify `x/action` state arrays.
2. Supernodes bootstrap an independent offline Kademlia DHT mesh on TCP `4445` to shard and retrieve encoded data.

## LEP-3: The Dual-Source Staking Model

To prevent Sybil attacks on the DA layer, Supernode registration enforces an aggregate minimum stake bound (`minimum_stake_for_sn`).

| Network | Target Stake Bound |
|---------|--------------|
| Testnet-2 | **10,000 LUME** (`10,000,000,000 ulume`) |

To encourage security compartmentalization, this bound is met by summing two distinct delegations:
1. **Validator Operator Delegation**: The native self-delegation bound to your consensus node.
2. **Supernode Wallet Delegation**: The dedicated wallet holding your `sn-manager` funds delegates to your validator.

*(Example: If your validator self-delegates 8,000 LUME, your Supernode wallet only needs to delegate 2,000 LUME).*

Verify the threshold dynamically:
```bash
lumerad query supernode params
```

## Hardware Thresholds

Due to the heavy RaptorQ encoding and massive I/O pipelines, Supernodes require elite infrastructure compared to standard Cosmos validators. 

| Component | Minimum | Production Target |
|-----------|---------|-------------|
| **Computing** | 8 Dedicated Cores (x86_64) | 16+ Cores |
| **Memory** | 16 GB RAM | 64 GB RAM |
| **Storage** | 1 TB NVMe SSD | 4 TB NVMe SSD |
| **Network** | 1 Gbps | 5 Gbps (Unmetered) |
| **OS** | Ubuntu 22.04 LTS | Ubuntu 24.04 LTS |

<Callout type="warn">
Standard HDDs or weak cloud VPS volumes will fail the I/O timeouts, automatically shifting your Supernode into a `POSTPONED` state. ARM processors are not currently supported by the `rq-library-wasm` binaries.
</Callout>

### Environment Port Bindings
Ensure your firewalls are strictly partitioned. 

**Supernode Machine:**
- `4444` (TCP): Supernode internal gRPC
- `4445` (TCP): Mesh P2P
- `8002` (TCP): External REST client gateway

**Validator Machine:**
- `9090` (TCP): Binding to `0.0.0.0` but strictly firewalled **ONLY** to the dedicated Supernode IP.

## Pre-Flight Checklist

- A fully synced and actively bonded validator node.
- A fully isolated bare-metal or high-tier VPS server instance.
- Safely exported `valoper` keys.
