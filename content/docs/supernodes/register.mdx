## Prerequisites

Before interacting with the `x/supernode` module, ensure:
- Your `sn-manager` daemon is initialized and running.
- You have safely stored the generic Supernode wallet address (`SN_ACCOUNT_ADDRESS`).
- Your core consensus validator (`valoper`) is actively bonded and synced.

---

## 1. Fund the Daemon Wallet

Because the `sn-manager` wallet will sign its own delegations, it must be funded with baseline gas and delegation tokens.

Run this transfer on your **Validator Host** (or any funded key):
```bash
lumerad tx bank send validator <SN_ACCOUNT_ADDRESS> 10000000000ulume \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume \
  -y
```
*Note: Ensure you send enough LUME to cover both the DA minimum stake threshold and baseline gas (e.g., `10000000000ulume` = 10,000 LUME).*

---

## 2. Execute LEP-3 Delegation

To satisfy the `minimum_stake_for_sn` threshold via the dual-source model, your Supernode daemon account must execute a native delegation transaction pointing to your Validator's operator address.

Run this transaction on any machine that retains access to the **Supernode keyring** (e.g., your Supernode server):
```bash
VALOPER="lumeravaloper1..."  # Your native consensus validator address

sn-manager tx staking delegate "$VALOPER" 10000000000ulume \
  --from supernode \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 7000ulume \
  -y
```

Verify your delegation is in state:
```bash
lumerad query staking delegations <SN_ACCOUNT_ADDRESS>
```

---

## 3. Broadcast `MsgRegisterSupernode`

<Callout type="warn">
**Validator Authorization Required**
Because a Supernode binds directly to a consensus node, the final registration transaction MUST be signed by your highly-secure **Validator keyring**. Run the following on your Validator machine.
</Callout>

```bash
# Extract your operator string
VALOPER=$(lumerad keys show validator --bech val -a)

# Define your public endpoints
SN_ENDPOINT="<SUPERNODE_PUBLIC_IP>:4444"
SN_ACCOUNT="lumera1..."  

# Broadcast the registration payload
lumerad tx supernode register-supernode \
  "$VALOPER" \
  "$SN_ENDPOINT" \
  "$SN_ACCOUNT" \
  --from validator \
  --chain-id lumera-testnet-2 \
  --gas auto --gas-adjustment 1.3 \
  --fees 5000ulume \
  -y
```

Await network confirmation. A successful response returns `code: 0`.

---

## 4. State Verification

### Consensus Engine Synchronization

On your **Validator server**, query the `x/supernode` state module:
```bash
lumerad query supernode get-supernode "$VALOPER"
```
Look for `state: SUPERNODE_STATE_ACTIVE`.

<Callout type="info">
If the chain returns a `SigningInfo not found` error immediately after broadcast, wait two blocks. The state machine requires block execution to append the node to the active DA array.
</Callout>

### Daemon Mesh Synchronization

On your **Supernode server**, execute a comprehensive health and routing check. Wait ~60 seconds after the `MsgRegisterSupernode` transaction to allow the daemon to ingest the block headers.
```bash
sn-manager check
```
